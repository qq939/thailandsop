# ROI 推理改造实施计划（多相机 / 多 ROI / 多模型）

## 1. 背景

当前系统以“整帧推理”为主。新需求是按 ROI（`x,y,w,h,a`）进行裁剪与旋转后推理，并将结果反算回整图显示与落库；同时支持每个 ROI 绑定不同模型，并保持多相机模式可扩展、可维护。

## 2. 目标

1. 支持每相机配置多个 ROI，ROI 参数为 `x,y,w,h,a`。
2. 支持 ROI 预览绘制、裁剪、旋转到水平后推理。
3. 支持推理结果从 ROI 局部坐标反变换回整图坐标。
4. 支持“每个 ROI 绑定一个模型”。
5. 支持多相机独立配置与独立运行，不互相影响。
6. 保持代码分层清晰，便于后续扩展新相机/新模型/新后处理。

## 3. 非目标（本阶段）

1. 不做旧 ROI 配置格式兼容迁移（按新格式直接落地）。
2. 不改数据库历史数据结构的向后兼容工具链。
3. 不引入独立进程拆分。

## 4. 现状基线（当前代码）

1. 推理与绘制主干在 `VideoPipeline`（取帧、推理、绘制、结果输出）。
2. 多相机是“每相机一个 `CameraSessionViewModel` + 一个 `VideoPipeline`”。
3. 配置体系已具备：`camera_config.json`（相机）、模型目录扫描（`DL/`）。

## 5. 设计原则

1. 单一职责：ROI 编辑、ROI 运行时处理、模型加载、结果落库分别解耦。
2. 数据驱动：运行逻辑依赖配置对象，不在 UI 内硬编码业务规则。
3. 可插拔：推理处理链抽象为处理器接口，保留整帧与 ROI 两种实现。
4. 多实例隔离：每个相机会话拥有独立 ROI 运行时上下文。
5. 性能优先：复用模型实例与图像缓存，避免每帧重复构造。

## 6. 目标架构

### 6.1 领域模型

建议新增：

- `RoiDefinition`
  - `Id`, `Name`, `Enabled`
  - `X`, `Y`, `W`, `H`, `A`（建议归一化坐标 + 角度）
  - `ModelId`
  - `DisplayColor`, `ZIndex`
- `CameraRoiPlan`
  - `CameraId`
  - `Enabled`
  - `List<RoiDefinition> Rois`
- `RoiSettings`
  - `Version`
  - `List<CameraRoiPlan> CameraPlans`

### 6.2 运行时组件

- `IFrameProcessor`（新抽象）
  - `bool TryProcess(FramePacket packet, out ProcessResult result)`
- `FullFrameProcessor`（整帧处理，兼容路径）
- `RoiFrameProcessor`（ROI 主处理器）
  - ROI 变换计划构建
  - ROI 裁剪旋转
  - ROI 内推理
  - 结果反变换回整图
  - ROI 绘制层
- `ModelRuntimePool`
  - 按 `modelId` 缓存并复用模型运行实例

### 6.3 数据流（目标）

1. 采集到原始帧。
2. 绘制 ROI 轮廓到预览层（可选开关）。
3. 对每个启用 ROI：
   - 按 `x,y,w,h,a` 计算仿射变换；
   - 裁剪并旋正 ROI 子图；
   - 用 ROI 绑定模型推理；
   - 将检测框/分割区域逆变换回整图坐标。
4. 合并所有 ROI 结果（可附加 `RoiId/ModelId` 元数据）。
5. 统一绘制到主画面并输出到结果管道（数据库/分析引擎/状态面板）。

## 7. 配置设计

新增配置文件：`roi_config.json`

```json
{
  "Version": 1,
  "CameraPlans": [
    {
      "CameraId": "camera-1",
      "Enabled": true,
      "Rois": [
        {
          "Id": "roi-1",
          "Name": "工位A",
          "Enabled": true,
          "X": 0.5,
          "Y": 0.5,
          "W": 0.3,
          "H": 0.2,
          "A": -12.0,
          "ModelId": "model-a",
          "DisplayColor": "#00BFA5",
          "ZIndex": 0
        }
      ]
    }
  ]
}
```

约定：

1. `X,Y,W,H` 使用归一化比例（相对原图宽高）。
2. `A` 使用角度（度），逆时针为正。
3. 某相机无 ROI 时可回退为“整帧 ROI”模式（不中断运行）。

## 8. UI 方案（ROI 配置窗口）

新增 `RoiConfigWindow`：

1. 左侧：图像画布（显示当前帧，支持 ROI 框选/拖拽/旋转）。
2. 右侧：ROI 列表 + 工具区 + 属性编辑区。
3. ROI 列表项字段：名称、启用状态、绑定模型、坐标摘要。
4. 工具按钮：新增、删除、复制、上移、下移、保存。
5. 属性区：`x,y,w,h,a`、模型下拉、颜色、启用开关。
6. 支持从当前相机抓一帧作为标定底图。

## 9. 实施阶段与里程碑

### 阶段 0：契约冻结

1. 冻结 ROI 坐标/角度语义。
2. 冻结 `roi_config.json` 结构。
3. 冻结 `RoiId/ModelId` 在结果对象中的字段约定。

验收：配置模型与示例文件可通过读写与校验。

### 阶段 1：配置层落地

1. 新增 `RoiSettings`、`RoiSettingsStorage`。
2. 在主流程加载 ROI 配置，并按 `CameraId` 分发到会话。

验收：配置可保存、重启后可恢复。

### 阶段 2：ROI 配置 UI

1. 新建 `RoiConfigWindow.xaml` + `RoiConfigViewModel`。
2. 实现 ROI 列表管理与属性编辑。
3. 接入模型列表（复用当前模型目录扫描结果）。

验收：可完成“新增 ROI -> 绑定模型 -> 保存 -> 重开可见”。

### 阶段 3：Pipeline 抽象改造

1. 在 `VideoPipeline` 引入 `IFrameProcessor`。
2. 先接入 `FullFrameProcessor`，保证行为不变。

验收：不启用 ROI 时，功能与性能与当前一致。

### 阶段 4：ROI 推理核心

1. 实现 `RoiFrameProcessor`。
2. 完成 ROI 仿射裁剪与旋正。
3. 完成结果逆变换回整图。
4. 实现多 ROI 结果聚合与冲突策略（先以直接合并为主）。

验收：已知 ROI 样例下，结果位置与人工标注一致（误差可控）。

### 阶段 5：多相机联动

1. 每个 `CameraSession` 注入自身 ROI 计划。
2. 运行中切换相机或配置更新时可热刷新（必要时重启会话）。

验收：2+ 相机同时运行，各自 ROI 与模型互不串扰。

### 阶段 6：结果与分析链路收口

1. 扩展结果实体附带 `RoiId/ModelId`。
2. 检查 `SqliteResultWriter`、`AnalysisEngine`、后续消费链适配。

验收：数据链路完整，UI/数据库可区分不同 ROI 结果来源。

### 阶段 7：性能与稳定性

1. 模型池复用、Mat 缓存复用。
2. 异常隔离：单 ROI 失败不拖垮整帧流程。
3. 增加诊断日志与关键耗时统计。

验收：达到产线目标 FPS，异常可追踪可恢复。

## 10. 测试策略

1. 几何正确性：构造可验证点位，校验正变换/逆变换。
2. 功能测试：单 ROI、多 ROI、单模型、多模型、模型缺失。
3. 多相机并发：会话隔离、配置隔离、结果隔离。
4. 回归测试：无 ROI 情况下行为与当前版本一致。
5. 压测：目标 FPS、内存占用、长时间运行稳定性。

## 11. 风险与应对

1. 风险：坐标系定义不统一导致结果偏移。
   - 应对：阶段 0 先冻结坐标契约，提供几何单测样例。
2. 风险：多模型并发导致显存占用突增。
   - 应对：模型池 + 按 ROI 复用 + 必要时串行/限流。
3. 风险：UI 与运行配置不同步。
   - 应对：统一配置源，保存后触发会话重载或版本刷新。
4. 风险：ROI 失败导致整帧中断。
   - 应对：ROI 粒度异常捕获，失败 ROI 跳过并记录日志。

## 12. 默认决策（本计划采用）

1. ROI 坐标使用归一化比例。
2. 角度 `A`：逆时针为正，单位为度。
3. 无 ROI 时自动回退为整帧处理。

## 13. 开工前检查清单

1. 是否确认 `roi_config.json` 字段命名与默认值。
2. 是否确认 ROI 配置窗口入口（主界面按钮位置）。
3. 是否确认结果表是否需要立即扩展 `RoiId/ModelId` 字段。
4. 是否确认第一阶段先只支持检测模型（后续再扩展序列模型）。
